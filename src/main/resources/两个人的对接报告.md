# 两个人的对接报告

本文档用于明确两个人并行开发时的职责边界和对接方式。

当前建议拆分为两条线：

- A 线：`PR6` 到 `PR9`
- B 线：`PR10` 到 `PR12`

其中：

- A 线负责把 `PR5` 的 review 上下文继续整理成可执行审查输入。
- B 线负责基于这个输入执行规则检查、调用 LLM，并最终回写 GitHub PR。

## 一、当前已经存在的上游输出

`PR5` 当前已经产出：

- `com.hdg.prysm.review.PrReviewContext`

它包含：

- `PrDiff diff`
- `List<PrReviewFileContext> files`

其中每个 `PrReviewFileContext` 包含：

- `PrChangedFile changedFile`
- `List<Snippet> snippets`
- `boolean truncated`
- `String note`

这意味着：

- `PR5` 已经完成 patch hunk 解析
- 已经从工作区提取出局部 snippet
- 已经处理 patch 缺失、删除文件、路径越界、符号链接、文件过大、总预算耗尽等基础边界

所以 `PR10` 到 `PR12` 不应该再重新解析 patch，也不应该再重新读取工作区文件。

## 二、两条线的职责边界

### A 线负责内容

- `PR6`：文件过滤与优先级排序
- `PR7`：上下文预算分配
- `PR8`：Review 输入组装与 Prompt 模板
- `PR9`：扩展上下文注入与上下文充足度门禁

A 线的职责是：

- 基于 `PrReviewContext` 过滤低价值文件
- 给文件排序
- 分配最终进入模型的上下文预算
- 组装统一执行输入
- 补充 PR title/body、少量 commit message、必要测试/接口信息
- 判断这次上下文是否足够支撑 AI 审查

### B 线负责内容

- `PR10`：规则引擎接入
- `PR11`：LLM Review 执行
- `PR12`：结果聚合与回写 GitHub PR

B 线的职责是：

- 接收统一执行输入
- 跑规则引擎
- 调用模型
- 合并规则结果和 AI 结果
- 组织最终评论并回写 GitHub PR

## 三、A 线交付给 B 线的唯一稳定入口

A 线最终必须交付一个稳定输入对象给 B 线。

建议名称：

- `ReviewExecutionInput`

这个对象是 `PR10`、`PR11`、`PR12` 的共同输入边界。

注意：

- B 线不要直接依赖 `PR6`、`PR7`、`PR8`、`PR9` 的内部实现细节。
- B 线只依赖 `ReviewExecutionInput` 的字段语义。

## 四、ReviewExecutionInput 建议结构

```java
public class ReviewExecutionInput {

    private final PrContext prContext;
    private final PrDiff diff;
    private final List<ReviewTargetFile> files;
    private final ContextStatus contextStatus;
    private final PromptPayload promptPayload;
}
```

字段含义：

- `prContext`
  - 当前审查对应哪个仓库、哪个 PR
  - 回写评论时也要用

- `diff`
  - 原始 PR diff
  - 保留文件级元数据和 patch

- `files`
  - 真正进入执行阶段的文件列表
  - 已经过滤、排序和预算裁剪

- `contextStatus`
  - 当前上下文状态
  - 包含 `FULL`、`LIMITED`、`INSUFFICIENT`、`SKIPPED`
  - `PR11` 根据它判断是否调用 LLM
  - `PR12` 根据它把降级原因写进评论

- `promptPayload`
  - 由 A 线最终组装出的 prompt 载荷
  - 包含 system prompt、user prompt、输出 schema
  - 这是给 `PR11` 直接使用的输入

### ContextStatus

```java
public class ContextStatus {

    private final ContextStatusCode code;
    private final String reason;
}
```

```java
public enum ContextStatusCode {
    FULL,
    LIMITED,
    INSUFFICIENT,
    SKIPPED
}
```

含义：

- `FULL`：上下文充足，可以正常审查
- `LIMITED`：上下文有限，可以审查但需要在结果中说明
- `INSUFFICIENT`：上下文不足，不建议调用 LLM 硬审
- `SKIPPED`：本次审查被跳过

### PromptPayload

```java
public class PromptPayload {

    private final String systemPrompt;
    private final String userPrompt;
    private final String outputSchema;
}
```

含义：

- `systemPrompt`：模型角色、审查原则、禁止编造等全局规则
- `userPrompt`：本次 PR 的实际上下文、patch、snippet 和元数据
- `outputSchema`：要求模型按什么 JSON 结构返回

## 五、ReviewTargetFile 建议结构

```java
public class ReviewTargetFile {

    private final PrChangedFile changedFile;
    private final List<PrReviewFileContext.Snippet> snippets;
    private final int priority;
    private final boolean selected;
    private final String note;
}
```

字段含义：

- `changedFile`
  - 原始 diff 文件信息
  - 包括 filename、status、additions、deletions、patch

- `snippets`
  - 从 `PrReviewContext` 中保留下来的局部上下文片段

- `priority`
  - 文件优先级
  - 数值越小表示越优先，或反之，但要在实现中固定规则

- `selected`
  - 是否最终进入执行阶段
  - 如果 A 线已经做完过滤和预算裁剪，这里通常应为 `true`

- `note`
  - 对这个文件的补充说明
  - 例如：
    - `patch unavailable`
    - `context truncated`
    - `filtered low value file`

## 六、B 线真正对接的到底是什么

B 线对接的不是：

- `PrReviewContextLoader`
- patch hunk 解析逻辑
- 本地文件读取逻辑
- PR6/PR7/PR8/PR9 的内部流程

B 线真正对接的是：

- `ReviewExecutionInput`

更具体一点：

- `PR10` 对接的是 `ReviewExecutionInput.files`
- `PR11` 对接的是 `ReviewExecutionInput.promptPayload`
- `PR12` 对接的是：
  - `ReviewExecutionInput`
  - `RuleEngineResult`
  - `LlmReviewResult`

所以 B 线看到的世界应该是：

```text
上游已经把可审查内容整理好了
我只负责执行规则检查、模型审查、结果回写
```

## 七、PR10、PR11、PR12 的输入输出边界

### PR10：规则引擎接入

输入：

- `ReviewExecutionInput`

输出：

- `RuleEngineResult`

职责：

- 读取需要审查的文件
- 跑 Semgrep / Checkstyle / SpotBugs 等规则
- 生成结构化规则结果

不负责：

- 不重新读取 patch
- 不重新读取工作区文件
- 不拼 prompt

### PR11：LLM Review 执行

输入：

- `ReviewExecutionInput`

输出：

- `LlmReviewResult`

职责：

- 基于 `promptPayload` 调用模型
- 获取结构化 AI findings

不负责：

- 不重新解析 diff
- 不自己做文件过滤和排序
- 不回写 GitHub PR

### PR12：结果聚合与回写 GitHub PR

输入：

- `ReviewExecutionInput`
- `RuleEngineResult`
- `LlmReviewResult`

输出：

- 最终评论内容
- GitHub 回写动作

职责：

- 合并规则结果和 AI 结果
- 去重
- 排序
- 组织 Markdown 评论
- 评论回 GitHub PR

## 八、统一 Finding 和结果结构

`PR10` 和 `PR11` 必须输出同一种 finding 结构，否则 `PR12` 无法稳定聚合、去重和排序。

### ReviewFinding

```java
public class ReviewFinding {

    private final String source;
    private final String severity;
    private final String filePath;
    private final Integer startLine;
    private final Integer endLine;
    private final String side;
    private final Integer line;
    private final String startSide;
    private final String title;
    private final String message;
    private final String suggestion;
    private final String ruleId;
}
```

字段含义：

- `source`
  - 发现来源，例如 `rule` 或 `llm`

- `severity`
  - 严重级别，例如 `error`、`warning`、`info`

- `filePath`
  - 问题所在文件路径
  - 为空时表示只能作为 summary finding 展示

- `startLine` / `endLine`
  - 问题在文件中的范围

- `side` / `line` / `startSide`
  - GitHub inline comment 定位字段
  - 常见取值为 `RIGHT` 或 `LEFT`
  - 如果缺失，`PR12` 可以降级成 summary comment

- `title`
  - 问题标题

- `message`
  - 问题说明

- `suggestion`
  - 建议修改方向

- `ruleId`
  - 规则 ID 或模型侧问题类型

### RuleEngineResult

```java
public class RuleEngineResult {

    private final List<ReviewFinding> findings;
    private final String summary;
}
```

### LlmReviewResult

```java
public class LlmReviewResult {

    private final List<ReviewFinding> findings;
    private final String summary;
    private final String rawResponse;
}
```

约定：

- `PR10` 和 `PR11` 都只产出 `ReviewFinding`
- `PR12` 不再适配两套结果结构
- `rawResponse` 只用于排查模型输出问题，不作为最终评论的主要内容

## 九、A 线和 B 线的禁止越界事项

### A 线不要做

- 不调用规则引擎
- 不调用模型
- 不回写 GitHub PR

### B 线不要做

- 不重新解析 hunk
- 不重新读取工作区文件
- 不重新做文件过滤、优先级和预算逻辑

## 十、对接完成标准

当下面公共契约落地后，两个人就可以稳定并行：

1. `ReviewExecutionInput`
2. `ReviewTargetFile`
3. `ContextStatus`
4. `PromptPayload`
5. `ReviewFinding`
6. `RuleEngineResult`
7. `LlmReviewResult`

做到这一步之后：

- A 线继续推进 `PR6` 到 `PR9`
- B 线可以独立推进 `PR10` 到 `PR12`

## 十一、当前推荐结论

当前项目里，`PR10` 到 `PR12` 可以交给另一个人连续负责。

但前提不是“拿到一堆零散 patch/snippet 就开干”，而是：

```text
A 线先交付稳定的 ReviewExecutionInput
B 线只围绕这个输入做执行、聚合和回写
```

这就是两个人最终的对接边界。
